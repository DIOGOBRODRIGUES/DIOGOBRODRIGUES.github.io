🔌 O que é uma API?

API = Application Programming Interface

"Uma API é um contrato de comunicação entre sistemas diferentes"

É como um garçom em um restaurante:

  • 🍽️ Você (cliente) faz um pedido
  • 👨‍🍳 A cozinha (servidor) prepara
  • 🤵 O garçom (API) entrega o resultado

📚 Tipos de APIs

🌐 REST

Usa HTTP e é stateless (sem estado)

🔄 GraphQL

Query language flexível para APIs

📡 SOAP

Protocol baseado em XML (legado)

⚡ WebSocket

Comunicação bidirecional em tempo real

🏛️ REST - Representational State Transfer

Criado por Roy Fielding em 2000

É um estilo arquitetural, não um protocolo

🎯 Princípios REST

  1. Cliente-Servidor: Separação de responsabilidades
  2. Stateless: Cada requisição é independente
  3. Cacheable: Respostas podem ser cacheadas
  4. Interface Uniforme: Padrões consistentes
  5. Sistema em Camadas: Arquitetura modular
  6. Código sob Demanda: (Opcional) Executar código no cliente

🌐 Métodos HTTP (Verbos)

🌐 Métodos HTTP (Verbos)

GET - Buscar/Ler recursos 
GET /users/123
POST - Criar novos recursos 
POST /users
PUT - Atualizar recursos completos 
PUT /users/123
PATCH - Atualizar parcialmente 
PATCH /users/123
DELETE - Remover recursos 
DELETE /users/123

📊 Códigos de Status HTTP

2xx - Sucesso ✅

  • 200 - OK
  • 201 - Created
  • 204 - No Content

3xx - Redirecionamento 🔄

  • 301 - Moved Permanently
  • 302 - Found (Temporary)
  • 304 - Not Modified

4xx - Erro do Cliente ❌

  • 400 - Bad Request
  • 401 - Unauthorized
  • 403 - Forbidden
  • 404 - Not Found

5xx - Erro do Servidor 💥

  • 500 - Internal Server Error
  • 502 - Bad Gateway
  • 503 - Service Unavailable

➡️ Método: POST

➡️ Endpoint: /api/login

➡️ Headers: Metadados da requisição

➡️ Body: Dados enviados (JSON)

🔗 Convenções de URLs RESTful

URLs devem ser substantivos, não verbos!

❌ Ruim


GET  /getUsers
POST /createNewUser
POST /deleteUser/123
GET  /getUserOrders/123

✅ Bom (RESTful)


GET    /users
POST   /users
DELETE /users/123
GET    /users/123/orders

🎯 Regras de Ouro

  • Use substantivos, não verbos
  • Use plural para coleções: /users
  • Use hierarquia: /users/123/orders
  • Use kebab-case: /order-items
  • Evite verbos nas URLs
  • Mantenha URLs simples e intuitivas

🔗 Endpoints Principais


# Autenticação
POST   /users         # Criar conta
POST   /login         # Login (retorna JWT)
GET    /me            # Dados do usuário logado

# Produtos
GET    /products      # Listar produtos (público)
POST   /products      # Criar produto (produtores)
PUT    /products/:id  # Atualizar produto
DELETE /products/:id  # Deletar produto

# Pedidos
POST   /order         # Criar pedido vazio
POST   /order/add     # Adicionar item ao pedido
GET    /order/:id     # Detalhes do pedido
PUT    /order/send    # Finalizar pedido
DELETE /order/remove  # Remover item

🏛️ Arquitetura: MVC + Services

Model-View-Controller (Adaptado)

📥 Controller
Recebe requisições HTTP, valida dados e chama o Service
⚙️ Service (Business Logic)
Contém a lógica de negócio, validações e regras da aplicação
💾 Model (Prisma)
Representa os dados e faz a comunicação com o banco de dados
🔒 Middleware
Intercepta requisições para autenticação, logs, validações, etc.

✨ Boas Práticas REST

  • ✅ Use substantivos no plural para recursos
  • ✅ Retorne códigos de status HTTP apropriados
  • ✅ Use JSON como formato padrão
  • ✅ Implemente versionamento: /api/v1/users
  • ✅ Use paginação para grandes listas
  • ✅ Documente sua API (Swagger/OpenAPI)
  • ✅ Implemente rate limiting
  • ✅ Use HTTPS sempre que possível
  • ✅ Valide todos os inputs
  • ✅ Nunca exponha senhas ou dados sensíveis
