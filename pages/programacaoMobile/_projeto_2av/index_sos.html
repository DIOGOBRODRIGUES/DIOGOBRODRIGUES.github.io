Reeguer 2.0 - Requisitos da Segunda Entrega

Evolução do aplicativo de auxílio a vítimas de desastres ambientais

Visão Geral

Para a segunda entrega, o aplicativo Reeguer deve evoluir de um protótipo para um produto mais completo, incorporando APIs externas, sensores do dispositivo, navegação avançada e experiência de usuário aprimorada.

Objetivo: Transformar o aplicativo Reeguer em uma plataforma robusta de auxílio a vítimas de desastres, com recursos de monitoramento em tempo real, mapeamento, notificações e coordenação de ajuda humanitária.

Requisitos Funcionais

1. Sistema de Navegação Avançado

  • Tab Navigation: Menu principal com abas para navegação entre seções principais (Home, Pedidos, Doações, Mapa, Perfil).
  • Stack Navigation: Navegação por pilha para formulários e detalhamento de informações.
  • Drawer Navigation: Menu lateral para configurações, informações, contatos de emergência e logout.
  • Deep Linking: Permitir acesso direto a áreas específicas do app via links externos.

2. Dashboard Interativo

  • Mapa de Incidentes: Visualização geográfica de áreas afetadas e pontos de ajuda.
  • Alertas em Tempo Real: Notificações sobre alertas meteorológicos e situações de risco.
  • Cards de Ação Rápida: Botões para reportar situação, solicitar ajuda e oferecer doações.
  • Estatísticas: Visão geral dos números de pessoas afetadas, pedidos atendidos e recursos mobilizados.

3. Monitoramento Meteorológico

  • Previsão do Tempo: Dados atualizados sobre condições climáticas locais.
  • Alertas de Precipitação: Notificações sobre chuvas intensas previstas.
  • Histórico Pluviométrico: Gráficos de volume de chuva nas últimas horas/dias.
  • Áreas de Risco: Mapeamento dinâmico de locais com risco de alagamento ou deslizamento.

4. Sistema de Pedidos de Ajuda Avançado

  • Geolocalização: Captura automática da localização do solicitante.
  • Categorização Detalhada: Especificação precisa das necessidades (tipo, quantidade, urgência).
  • Anexo de Mídia: Possibilidade de anexar fotos da situação.
  • Status e Acompanhamento: Sistema de acompanhamento do estado do pedido.

5. Sistema de Doações e Voluntariado

  • Catálogo de Necessidades: Lista filtrada por tipo, localização e urgência.
  • Match de Doações: Relacionamento entre ofertas e necessidades.
  • Logística: Coordenação de pontos de coleta e entrega.
  • Feedback: Sistema para confirmar recebimento e adequação das doações.

6. Integração com APIs Externas

  • API Meteorológica: Integração com serviços como OpenWeatherMap ou Weather API.
  • Serviços de Mapeamento: Google Maps ou Mapbox para visualização geográfica.
  • Alertas Governamentais: API de órgãos como Defesa Civil ou serviços equivalentes.
  • Compartilhamento Social: Integração com redes sociais para divulgação de campanhas.

7. Uso de Sensores do Dispositivo

  • Geolocalização: GPS para identificar posição do usuário e proximidade com áreas de risco.
  • Câmera: Captura de imagens para documentar situações e necessidades.
  • Acelerômetro: Detecção de movimentos bruscos que possam indicar deslizamentos.
  • Microfone: Gravação opcional de mensagens de voz para pedidos de socorro.

8. Sistema de Abrigos e Pontos Seguros

  • Mapa de Abrigos: Localização geo-referenciada de todos os abrigos disponíveis.
  • Informações Detalhadas: Capacidade, ocupação atual, infraestrutura e necessidades.
  • Rotas de Evacuação: Trajetos seguros até o abrigo mais próximo.
  • Check-in/Check-out: Sistema para registro de presença nos abrigos.

Requisitos Não-Funcionais

1. Experiência do Usuário

  • Design Responsivo: Adaptação para diferentes tamanhos de tela usando useWindowDimensions.
  • Feedback Visual: Indicadores de carregamento, sucesso e erro em todas as operações.
  • Acessibilidade: Suporte para leitores de tela e navegação alternativa.
  • Modo Offline: Funcionalidade básica mesmo sem conexão à internet.

2. Gerenciamento de Estado

  • Context API: Gerenciamento de estado global do aplicativo.
  • Armazenamento Local: Persistência de dados com AsyncStorage.
  • Gestão de Cache: Armazenamento local de dados frequentemente acessados.

3. Performance e Otimização

  • Renderização Eficiente: Uso de FlatList com otimizações para grandes conjuntos de dados.
  • Lazy Loading: Carregamento sob demanda para conteúdos extensos.
  • Gerenciamento de Memória: Evitar memory leaks com cancelamento adequado de subscrições.

4. Segurança

  • Proteção de Dados Pessoais: Criptografia de informações sensíveis.
  • Autenticação Segura: Sistema de login com validação adequada.
  • Permissões: Solicitação explícita e justificada para acesso a sensores do dispositivo.

Escopo Mínimo (Requisitos Obrigatórios)

Para aprovação, o projeto deve implementar, no mínimo:

  1. Sistema de navegação completo (Tab + Stack) entre todas as telas principais.
  2. Integração com pelo menos uma API externa (meteorológica ou de mapas).
  3. Uso de pelo menos dois sensores do dispositivo (GPS e câmera).
  4. FlatList otimizada para exibição de pedidos de ajuda e doações.
  5. Persistência de dados com AsyncStorage.
  6. Interface responsiva e consistente em diversos dispositivos.
